srsRAN 5G NR jBPF Telemetry Pipeline — Project Summary & Current State
======================================================================

Project Overview
----------------
We built a complete 5G NR MAC-layer telemetry pipeline using srsRAN Project
(with jBPF hooks), a custom ZMQ channel broker for RF impairment simulation,
and a Grafana + InfluxDB dashboard for real-time visualization. The system
runs entirely on a single Ubuntu 22.04 machine (user maxim, sudo password
2003) using ZMQ virtual radio (no physical RF hardware).


Architecture
------------
jrtc (runtime controller)
  -> ZMQ Channel Broker (Rician fading + AWGN noise injection)
    OR: GNU Radio Channel Broker (frequency-selective fading + AWGN + CFO + SRO)
    -> gNB (5G base station with ~60 jBPF eBPF codelets across MAC/RLC/PDCP/FAPI/RRC/NGAP/UE_CTX)
      -> Reverse Proxy (IPC->TCP bridge, port 30450)
      -> Decoder (protobuf->JSON, gRPC:20789, UDP:20788)
        -> telemetry_to_influxdb.py (Python ingestor, parses JSON -> InfluxDB)
          -> InfluxDB 1.6.7 (port 8086, db=srsran_telemetry, 15 measurements)
            -> Grafana 11.5.2 (port 3000, admin/admin, 39-panel dashboard)
  -> srsUE (UE simulator, ports 2000/2001)
    -> iperf3 (UDP uplink traffic generator, 10M for 3600s)

The ZMQ broker sits between gNB (ports 4000/4001) and UE (ports 2000/2001),
intercepting IQ samples and injecting AWGN noise + Rician flat fading before
forwarding.

What We Built (Chronological)
-----------------------------

Phase 1: ZMQ Channel Broker (from scratch)

  Problem: srsRAN's ZMQ virtual radio creates a perfect channel (zero noise,
  zero fading), producing flat/boring telemetry (constant SINR=42 dB, MCS=28,
  0 HARQ retx).

  Solution: Wrote zmq_channel_broker.c — a standalone C program that
  intercepts IQ samples between gNB and UE at the ZMQ transport layer.

  AWGN: Additive White Gaussian Noise via Box-Muller transform. Per-subframe
  RMS power estimation, configurable SNR.

  Fading (originally Rayleigh): AR1 model based on Jake's/Clarke's Doppler
  spectrum. alpha = J0(2*pi*f_d*T) correlation, complex multiply on IQ pairs.
  Noise computed from pre-fading signal power so deep fades cause real SNR
  drops.

  Bugs fixed: ZMQ REQ-REP state machine violation (EFSM error), SIGPIPE/EINTR
  handling, port conflicts (3001->4001), process isolation via setsid.


Phase 2: Rayleigh -> Rician Fading Conversion

  Problem: Pure Rayleigh fading caused complete channel nulls (infinite fade
  depth), crashing the UE within 1-3 minutes via PBCH sync loss.

  Solution: Added Line-of-Sight (LoS) component to the fading model,
  converting from Rayleigh to Rician. Channel gain = sqrt(K/(K+1)) (LoS) +
  sqrt(1/(K+1)) * h_rayleigh (scatter).

  K-factor: Controls LoS/scatter ratio. K=0 -> pure Rayleigh, K->inf ->
  AWGN-like. Specified in dB via --k-factor. Added --rayleigh shortcut
  (K=-100 dB). This bounds fade depth: K=6 dB limits worst-case fades to
  ~10-15 dB (vs unlimited for Rayleigh).


Phase 3: Multi-Codelet Expansion

  Expanded from 1 codelet set (MAC, 10 codelets) to 11 codelet sets (~60
  codelets) covering:

    MAC:  CRC stats, BSR, UCI/CQI, DL HARQ, UL HARQ (10 codelets)
    RLC:  DL/UL SDU/PDU stats (11 codelets)
    PDCP: DL/UL PDU/SDU stats (10 codelets)
    FAPI: DL config, UL config, CRC, RACH (4x2 codelets)
    RRC:  UE add/remove/procedure/update (5 codelets)
    NGAP: Procedure started/completed/reset (3 codelets)
    perf: jBPF hook timing stats (1 codelet)

  Modified all YAML configs from DestinationNone to DestinationUDP. 10/11
  sets load successfully; ue_contexts always fails (missing compiled .o file)
  — this is expected. Pipeline produces 17 telemetry schemas across the full
  5G NR protocol stack.


Phase 4: Launch & Stop Scripts

  launch_mac_telemetry.sh: Starts all 8 components in correct order with
  dependency checks, process health verification, colored output, and
  comprehensive CLI flags.

  stop_mac_telemetry.sh: Reverse-order teardown with sudo kill for privileged
  processes.


Phase 5: Grafana + InfluxDB Telemetry Dashboard

  InfluxDB 1.6.7: Database srsran_telemetry with 15 measurements
  (mac_crc_stats, mac_harq_stats, mac_bsr_stats, mac_uci_stats,
  rlc_dl_stats, rlc_ul_stats, pdcp_dl_stats, pdcp_ul_stats,
  fapi_dl_config_stats, fapi_ul_config_stats, fapi_crc_stats,
  fapi_rach_stats, rrc_events, ngap_events, jbpf_perf_stats).

  Python ingestor (telemetry_to_influxdb.py): Tails /tmp/decoder.log, parses
  JSON, maps 16 proto schemas to 15 measurements, handles cumulative->delta
  for MAC counters, writes via InfluxDB HTTP line protocol.

  Grafana dashboard: 39 panels organized in rows by protocol layer. Includes
  time-series, stat panels, gauges, and bar charts. Auto-provisioned via
  grafana/provisioning/.

  Fixed sparse data panels: PDCP DL (event-driven, widened to 1h window), RRC
  Events and NGAP Events (widened to 24h window) — these panels showed "no
  data" because their event-driven telemetry aged out of the default 5-minute
  $timeFilter.


Phase 6: HARQ Failure Parameter Tuning

  Goal: Find fading parameters that trigger HARQ failures (observable in
  telemetry) without crashing the radio connection.

  Challenge: Parameters aggressive enough for failures also tend to crash the
  RRC layer ("RRC container not ACKed within 120msec" -> UE context release).

  Systematic testing of 6 configurations:

    K=4, SNR=25, fd=5  ->  ~4 min   -> Crashed
    K=4, SNR=28, fd=5  ->  ~1 min   -> Crashed
    K=6, SNR=25, fd=5  ->  ~1.5 min -> Crashed
    K=6, SNR=30, fd=5  ->  5+ min   -> Stable, 0 failures
    K=3, SNR=30, fd=5  ->  7+ min   -> Stable, 0 failures
    K=3, SNR=28, fd=5  ->  15+ min  -> 12 HARQ failures (WINNER)

  Winning config: K=3 dB, SNR=28 dB, fd=5 Hz — produces 12 HARQ failures
  across 2 deep fade events (min_sinr=-36 dB, tx_success_rate dips to 76%),
  max_retx=4 (all retries exhausted), connection stable 15+ minutes.

  Updated launch script defaults: ZMQ_BROKER_SNR=28, ZMQ_BROKER_K_FACTOR=3.


Phase 7: Comprehensive Plotting Pipeline

  Created plot_all_telemetry.py (944 lines) — generates 15 PNGs covering all
  17 telemetry schemas. One plot per codelet schema, strictly codelet data
  only (no synthetic/derived views).

  Key implementation details:
  - Parses decoder log via regex + JSON extraction
  - Disambiguates MAC vs FAPI crc_stats by _schema_proto_package field
  - Divides FAPI "Avg" cumulative sums by l1Cnt for true averages
  - Aggregates RLC/PDCP per-bearer stats across all bearers
  - Filters RNTI > 1000 for UE data (vs system/SIB RNTIs)
  - Event timelines for RRC/NGAP (few data points per session)
  - matplotlib Agg backend, 150 DPI, outputs to ~/Desktop/plots/

    SINR mean:           30.3 dB (matches configured SNR)
    SINR std dev:        4.6 dB (Rayleigh fading signature)
    SINR range:          17.6–47.7 dB (30 dB spread, realistic)
    Deep fades:          9.9% of samples >6 dB below mean

    TX success rate:     95.9% mean, 33.3% worst window
    DL MCS:              19.7–27.9 (mean 26.9, adaptive modulation working)
    UL MCS:              18.6–28.0 (mean 27.1)
    CQI:                 15 constant (expected — UE-computed)
    BSR buffer:          76 bytes – 3.8 MB (builds up during fades)
    RRC lifecycle:       1 add → 3 procedures → 1 remove (correct)
    NGAP lifecycle:      3 started → 3 completed, all success
    jBPF hook latency:   p50=192–384 ns, p99<1.5 µs (negligible overhead)

  DL-side plots are mostly empty/flat — correct because we ran uplink-only

Key Technical Details
---------------------
- Telemetry flow: 11 codelet sets -> 16 proto schema types -> 15 InfluxDB
  measurements -> 39 Grafana panels

- Channel broker options:
  1. C broker (zmq_channel_broker) — flat Rician/Rayleigh fading + AWGN
  2. GNU Radio broker (grc_channel_broker.py) — frequency-selective fading
     (EPA/EVA/ETU), AWGN, CFO, SRO. Use --grc flag in launch script.

- Known issue: slice_mgmt codelet cannot compile — jbpf_slice_allocation type
  not in this srsRAN version. Not loaded in pipeline. 2 ue_contexts codelets
  disabled (jbpf_pdu_session_ctx_info missing). All other 57 codelets work.

- HARQ failure mechanism: At K=3 + SNR=28, DL fades reach -53.4 dB. Effective
  SINR drops to -36 dB during worst fades, below the LDPC decoding threshold
  (~15-18 dB). All 4 HARQ retransmission attempts are exhausted -> MAC
  declares failure.

- Why crashes happen with lower K/SNR: Sustained deep fades (>120ms) trigger
  gNB's RRC timeout -> UE context release -> connection drops. The Rician LoS
  component (K=3) keeps nulls brief enough to avoid this.
